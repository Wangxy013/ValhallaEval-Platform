# 测试驱动开发

## 适用场景

功能开发、缺陷修复、重构和行为变更默认遵循测试驱动开发。纯文档、纯配置或纯工具适配文件改动可以例外，但必须明确说明原因。

## 必须遵循

1. 先写最小失败测试，准确描述待实现或待修复的行为。
2. 运行最小相关测试目标，确认失败原因符合预期。
3. 实现让测试通过所需的最小生产代码。
4. 重新执行相关测试，确认通过。
5. 仅在保持绿灯状态下进行必要重构。

## 仓库要求

- 前端纯逻辑优先补充 Node 测试，并保持与现有 `helpers.test.ts` / `csv.test.ts` 风格一致。
- 后端优先在对应模块附近补 Rust 单元测试；若行为跨模块或跨路由，可补集成测试。
- 任务创建相关改动必须覆盖三阶段模型约束，不允许只靠手工验证代替测试。
- 任何涉及 API Key 掩码或更新逻辑的改动，都应优先补自动化回归。
